# IToguchi（いとぐち）要件定義書 Ver 4.0

## 1. 製品・サービス概要

- **製品名**: IToguchi（いとぐち）
- **コンセプト**: 「個人の店主が『LINEひとつで』予約・会員証・接客を自動化し、本業に集中できる環境を作るSaaS型予約管理システム」
- **提供価値**:
  - 開発技術を持たない店舗オーナーが、自ら DX（業務効率化）を推進できる環境。
  - GAS を一切使用しない堅牢なデータ管理と、モダンな操作感の提供。

## 2. システムアーキテクチャ

フロントエンド（Vercel）とバックエンド（Supabase）を完全に分離し、スケーラビリティと高速なレスポンスを両立。

| レイヤー            | 採用技術                    | 役割                                                     |
| :------------------ | :-------------------------- | :------------------------------------------------------- |
| **フロントエンド**  | React (Vite) / Tailwind CSS | 管理画面 UI、設定フォーム、データ可視化（Recharts 等）   |
| **ホスティング**    | Vercel                      | フロントエンド配信、GitHub 連携による CI/CD              |
| **バックエンド/DB** | Supabase                    | 認証(Auth)、PostgreSQL、Realtime 通信、ストレージ        |
| **Webhook 処理**    | Supabase Edge Functions     | LINE からの信号受信、OpenAI 連携、ビジネスロジックの実行 |
| **外部連携**        | LINE API / OpenAI API       | Messaging API / LIFF、AI 対話生成                        |

## 3. データベース物理設計（Supabase/PostgreSQL）

マルチテナント（複数顧客管理）に対応し、データ整合性を重視。

- **profiles**: 管理者（ユーザー）情報、プラン（free/pro）、契約ステータス。
- **line_accounts**: Messaging API キー、アクセストークン (`channel_access_token`)、bot_id（ユニーク URL 用）、**line_user_id**（Webhook 宛先識別用）。
- **auto_responses**: キーワード応答ルール（キーワード、返信文、AI 優先フラグ）。
- **reservations**: 予約データ（日時、LINE ユーザー ID、ステータス、店舗側メモ、**google_event_id**）。
- **google_calendar_settings**: Google カレンダー連携設定（リフレッシュトークン、同期対象カレンダー ID）。
- **points**: 顧客別ポイント管理（現在のポイント数、最終更新日、ポイント履歴 URL）。
- **customer_logs**: LINE ユーザーとの接触履歴、送信メッセージログ。

## 4. 機能詳細

### 4.1 Web 管理画面（React UI）

GAS による管理を廃止し、すべての操作を Web UI 上で行う。

- **統合ダッシュボード**: 友だち数、予約状況、ポイント発行推移のグラフ表示。
- **CRM（顧客管理）**: スプレッドシートを介さず、ブラウザ上で直接顧客情報を検索・編集。
- **応答設定エディタ**: リスト形式での応答ルール作成。プレビュー機能付き。
- **予約・ポイント操作**: 管理者が手動で予約を追加・変更したり、ポイントを付与する機能。
- **CSV エクスポート**: 解析用にデータをスプレッドシートや Excel で扱いたいユーザー向け。

### 4.2 LINE ボット機能（Edge Functions）

- **高速 Webhook 応答**: Edge Functions (Deno) による低遅延レスポンス。Node.js 依存ライブラリを排除し、Web Crypto API で実装。
- **マルチテナント識別**: Webhook リクエスト内の `destination` (LINE User ID) を基に、適切な店舗アカウントを自動判別。
- **予約・ポイント LIFF**: LINE 内で開く高機能な専用 UI。予約カレンダーやマイページを提供。
- **AI 自動接客**: 既存の応答ルールに該当しない場合、OpenAI が店舗情報を基に回答。

### 4.3 予約同期システム（Google カレンダー連携）

- **アーキテクチャ**: Google カレンダーを「予約のハブ」として利用し、ホットペッパー等の外部予約システムとのダブルブッキングを防止。
- **OAuth 連携**: `access_type=offline` によりリフレッシュトークンを取得し、永続的なバックグラウンド同期を実現。
- **リアルタイム同期 (Watch)**: Google Calendar API の Webhook 機能を利用し、カレンダーの変更を即座に検知して在庫を更新。
- **空き枠計算**: ユーザーが予約画面を開いた瞬間に、IToguchi 内の予約と Google カレンダーの予定をマージして正確な空き時間を提示。

## 5. セキュリティ・運用方針

- **Row Level Security (RLS)**: Supabase の RLS により、**「店舗 A のデータは、店舗 A の管理者のみが読み書きできる」**ことを DB レベルで強制。
- **API 保護**: LINE Webhook の署名検証を Edge Functions 内で厳格に実施。
- **環境変数管理**: Vercel/Supabase の環境変数機能を使い、認証キーを安全に保持。

## 8. エグゼクティブプランと独自開発の運用

ボトルネックを防ぎつつ、開発者としての単価を最大化する設計です。

- **独自開発の導線**: プロプランのユーザーが「もっとこだわりたい」となった際、エグゼクティブプランへアップグレードすることで、初めて個別見積もりによる独自開発を請け負う。
- **保守**: 独自機能は機能フラグで管理し、ベースのシステムは一つに保つことで保守コストを抑制。
- **個別相談**: Calendly ＋ Stripeを連携させ、予約と同時に決済が完了する「事前決済制」で自動化。

## 9. 法的・事務的基盤

副業としてのプライバシーと信頼性を両立させた体制です。

- **特定商取引法に基づく表記**:
  - 住所：バーチャルオフィス（GMOやDMM等）を活用。
  - 電話：050番号（IP電話アプリ）を活用。
- **決済システム**: Stripeを活用し、サブスク（月額）、単発（代行費）、請求書（独自開発費）をすべて一元管理。
